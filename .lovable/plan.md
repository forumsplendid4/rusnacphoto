
Цель: одновременно исправить критичные сбои (заказ/удаление), вернуть явное редактирование фото в мероприятии и добавить управляемую загрузку с прогрессом/ETA/отменой.

Что я нашел по текущему состоянию:
- В `AdminDashboard` кнопка управления фото технически есть (`Управлять фото`), но её сделаю более явной как «Редактировать фото» и закреплю в видимой группе действий.
- В логах БД есть ошибки `new row violates row-level security policy for table "orders"` — это объясняет, почему часть заказов не проходит (особенно если где-то используется старый клиентский поток).
- Для удаления мероприятий явных DB-ошибок в логах нет, поэтому проблему закрою через более надежный клиентский вызов (try/catch + retry + понятные ошибки), чтобы не “падало молча” на сетевых сбоях.
- В загрузке фото сейчас нет ни прогресса, ни ETA, ни отмены.

План реализации:

1) Надежный слой вызовов backend RPC (устранение “не удаляется/не отправляется”)
- Добавить универсальный helper для RPC с:
  - `try/catch`,
  - 2–3 ретрая на сетевые ошибки (`Load failed`, `Failed to fetch`),
  - нормализацией текста ошибок для UI.
- Перевести критичные вызовы на этот helper:
  - `create_order_with_items` (отправка заказа),
  - `admin_delete_event` (удаление мероприятия),
  - `admin_get_events` (загрузка админки, чтобы не “исчезали” кнопки при временном сбое).
- В `CartDrawer` показать отдельные сообщения для:
  - сетевой ошибки,
  - неактивного мероприятия,
  - невалидных позиций корзины.

2) Исправление RLS для стабильной отправки заказов
- Добавить миграцию, которая явно разрешает `INSERT` в `orders` и `order_items` для публичного/авторизованного клиента, не открывая чтение:
  - `INSERT` разрешен,
  - `SELECT/UPDATE/DELETE` остаются закрыты.
- Это даст совместимость и с текущим RPC-потоком, и со старыми клиентскими сессиями/кэшированными сборками, из-за которых и появляется ошибка RLS в логах.

3) Шкала загрузки фото + ETA + отмена с подтверждением
- В `AdminDashboard` добавить состояние загрузки:
  - `total`, `processed`, `success`, `failed`,
  - `startedAt`,
  - `isUploading`,
  - `cancelRequested`.
- Добавить визуальный блок прогресса (на основе `Progress`):
  - процент,
  - текст `X / Y`,
  - примерное оставшееся время (`ETA`) по формуле:
    - `avg = elapsed / processed`,
    - `remaining = avg * (total - processed)`.
- Добавить кнопку-крестик внутри блока прогресса:
  - по нажатию открывать `AlertDialog` с подтверждением:
    - «ТОЧНО ХОТИТЕ ОТМЕНИТЬ ЗАГРУЗКУ?»
  - при подтверждении:
    - ставить `cancelRequested = true`,
    - останавливать взятие новых файлов в воркерах,
    - показывать финальный статус: сколько успело загрузиться.
- Поведение отмены: уже загруженные файлы остаются; отменяются только оставшиеся в очереди.

4) Возврат и усиление кнопки редактирования фото мероприятия
- Явно вернуть CTA в карточке мероприятия как:
  - «Редактировать фото» (или «Редактировать мероприятие» + подпись про фото),
  - стабильная позиция рядом с «Заказы».
- Убедиться, что `EventPhotosManagerDialog` открывается для выбранного события и после удаления фото корректно обновляет счетчики.

5) Проверка результата (после внедрения)
- Админ:
  - загрузка 20+ фото → видно прогресс, ETA, работает отмена по крестику с подтверждением.
  - удаление мероприятия проходит с первого раза или корректно ретраится/показывает понятную ошибку.
  - кнопка «Редактировать фото» видима и работает.
- Клиент:
  - заказ из корзины отправляется, запись появляется в админке.
- Регрессии:
  - Excel-экспорт заказов не ломается,
  - группировка заказов по клиенту сохраняется.
